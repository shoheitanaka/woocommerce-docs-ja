# Deprecation in core

非推奨（Deprecation）とは、後方互換性を破壊したり、その使用を完全に禁止したりすることなく、ある機能や慣習の使用を控え、他のものを優先させる方法である。非推奨に関するウィキペディアの記事を引用する：

> 非推奨のソフトウェア機能がソフトウェアに残っている間、その機能を使用すると、代替方法を推奨する警告メッセージが表示されることがあります。非推奨のステータスは、その機能が将来削除されることを示すこともあります。非推奨のステータスは、その機能が将来削除されることを示す場合もある。機能はすぐに削除されるのではなく、後方互換性を提供し、プログラマに影響を受けるコードを新しい標準に準拠させる時間を与えるために、非推奨となる。

関数、メソッド、機能が非推奨になる理由はさまざまです。いくつか挙げてみよう：

- もう使わない関数を削除したいかもしれない。
- ある関数の動作を変更または改善することに決めたが、後方互換性を壊すため、代わりに古い関数を非推奨にする必要があるかもしれない。
- 命名規則を標準化したいかもしれない。
- 同じコードを別の場所で再利用するのを避けるために、似たような関数をマージする機会を見つけるかもしれない。

非推奨告知は理想的でも魅力的でもないが、単なる警告であり、エラーではない。 

非推奨の警告は、あなたのストアが壊れているという意味ではありません。

## How do we deprecate functions?

WooCommerceで何かを非推奨にする場合、開発者に明確にし、後方互換性を維持するためにいくつかのアクションを取ります。

1.関数やメソッドに、その関数がどのバージョンで非推奨になったかを示すdocblockを追加します（`@deprecated 2.x.x`など）。
2.独自の`wc_deprecated_function`関数を使って、どのバージョンで、どの関数で、どのような置き換えが可能かを示す警告表示を追加します。これについてはもう少し詳しく説明する。
3.コードベース全体を通して、非推奨関数の使用を削除します。

関数やメソッド自体はコードベースから削除されない。これにより、削除されるまで（通常は1年以上先か、数回先のメジャーリリースまで）後方互換性が保たれます。

上記で`wc_deprecated_function`について触れましたが、これは`_deprecated_function`WordPress関数の独自のラッパーです。この関数は、AJAXイベント中に`WP_DEBUG`の値に関係なく、ログを表示する代わりにログエントリを強制的に生成することを除けば、非常によく似た動作をします。

## 非推奨関数が呼ばれたらどうなるか？

拡張機能やテーマが非推奨の関数を使用している場合、次の例のような警告が表示されることがあります：

```bash
Notice: woocommerce_show_messages is deprecated since version 2.1! Use wc_print_notices instead. in /srv/www/wordpress-default/wp-includes/functions.php on line 3783
```

これは、いつから、どこで、何が非推奨となり、どのような代替手段があるのかを教えてくれる。

通知や警告は通常インラインで表示されますが、それらを集めてサイトのフッターにきれいに表示するプラグインもあります。例えば、[Query Monitor](https://wordpress.org/plugins/query-monitor/)を考えてみましょう。

### 本番での警告（ストアオーナーの方は必ずお読みください）

PHPの通知や警告(あるいはその他のエラー)を本番のストアに表示することは非常に推奨されません。これらは、悪意のあるユーザがあなたのストアにアクセスするために悪用する可能性のある、あなたのセットアップに関する情報を明らかにする可能性があります。これらは公開されないようにし、オプションでログに記録するようにしてください。

WordPressでは、`wp-config.php`にいくつかの定数を追加または変更することでこれを行うことができます：

```php
define( 'WP_DEBUG', false );
```

ホストによっては、ホストの設定によりエラーが表示される場合があります。強制的にエラーを表示させないようにするには、`wp-config.php`にもこれを追加する必要があるかもしれない：

```php
@ini_set( 'display_errors', 0 );
```

通知を表示する代わりにログに記録するには、次のようにする：

```php
define( 'WP_DEBUG', true );
define( 'WP_DEBUG_LOG', true );
define( 'WP_DEBUG_DISPLAY', false );
```

WordPressのエラーログのデフォルトの場所は`wp-content/debug.log`です。

このログは一般に公開される可能性があり、セキュリティ上のリスクもあることに注意してください。非公開にするには、プラグインを使うか、`wp-config.php`でカスタムパスを定義します。

```php
<?php
/**
 * Plugin Name: Custom Debug Log Path
 */

ini_set( 'error_log', '/path/to/site/logs/debug.log' );
```
